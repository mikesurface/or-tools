00017 // An implementation (with some improvements) of the star-representation of a
00018 // graph as described in J. Ebert, "A versatile data structure for
00019 // edge-oriented graph algorithms." Communications of the ACM 30(6):513-519
00020 // (June 1987). http://portal.acm.org/citation.cfm?id=214769
00021 
00022 // In this file there are two representations that have much in
00023 // common. The general one, called simply EbertGraph, contains both
00024 // forward- and backward-star representations. The other, called
00025 // ForwardEbertGraph, contains only the forward-star representation of
00026 // the graph, and is appropriate for applications where the reverse
00027 // arcs are not needed.
00028 
00029 // In the general EbertGraph case, the graph is represented with three
00030 // arrays.
00031 // Let n be the number of nodes and m be the number of arcs.
00032 // Let i be an integer in [0..m-1], denoting the index of an arc.
00033 //  * head_[i] contains the end-node of arc i,
00034 //  * head_[-i-1] contains the start-node of arc i.
00035 // Note that in two's-complement arithmetic, -i-1 = ~i.
00036 // Consequently:
00037 //  * head_[~i] contains the start-node of the arc reverse to arc i,
00038 //  * head_[i] contains the end-node of the arc reverse to arc i.
00039 //  Note that if arc (u, v) is defined, then the data structure also stores
00040 //  (v, u).
00041 //  Arc ~i thus denotes the arc reverse to arc i.
00042 //  This is what makes this representation useful for undirected graphs and for
00043 //  implementing algorithms like bidirectional shortest paths.
00044 //  Also note that the representation handles multi-graphs. If several arcs
00045 //  going from node u to node v are added to the graph, they will be handled as
00046 //  separate arcs.
00047 //
00048 // Now, for an integer u in [0..n-1] denoting the index of a node:
00049 //  * first_incident_arc_[u] denotes the first arc in the adjacency list of u.
00050 //  * going from an arc i, the adjacency list can be traversed using
00051 //    j = next_adjacent_arc_[i].
00052 //
00053 // This implementation has the following benefits:
00054 //  * It is able to handle both directed or undirected graphs.
00055 //  * Being based on indices, it is easily serializable. Only the contents
00056 //    of the head_ array need to be stored. Even so, serialization is
00057 //    currently not implemented.
00058 //  * The node indices and arc indices can be stored in 32 bits, while
00059 //    still allowing to go a bit further than the 4-gigabyte
00060 //    limitation (48 gigabytes for a pure graph, without capacities or
00061 //    costs.)
00062 //  * The representation can be recomputed if edges have been loaded from
00063 //  * The representation can be recomputed if edges have been loaded from
00064 //    external memory or if edges have been re-ordered.
00065 //  * The memory consumption is: 2 * m * sizeof(NodeIndexType)
00066 //                             + 2 * m * sizeof(ArcIndexType)
00067 //                             + n * sizeof(ArcIndexType)
00068 //    plus a small constant.
00069 //
00070 // This implementation differs from the implementation described in [Ebert 1987]
00071 // in the following respects:
00072 //  * arcs are represented using an (i, ~i) approach, whereas Ebert used
00073 //    (i, -i). Indices for direct arcs thus start at 0, in a fashion that is
00074 //    compatible with the index numbering in C and C++. Note that we also tested
00075 //    a (2*i, 2*i+1) storage pattern, which did not show any speed benefit, and
00076 //    made the use of the API much more difficult.
00077 //  * because of this, the 'nil' values for nodes and arcs are not 0, as Ebert
00078 //    first described. The value for the 'nil' node is set to -1, while the
00079 //    value for the 'nil' arc is set to the smallest integer representable with
00080 //    ArcIndexSize bytes.
00081 //  * it is possible to add arcs to the graph, with AddArc, in a much simpler
00082 //    way than described by Ebert.
00083 //  * TODO(user) although it is already possible, using the
00084 //    GroupForwardArcsByFunctor method, to group all the outgoing (resp.
00085 //    incoming) arcs of a node, the iterator logic could still be improved to
00086 //    allow traversing the outgoing (resp. incoming) arcs in O(out_degree(node))
00087 //    (resp. O(in_degree(node))) instead of O(degree(node)).
00088 //  * TODO(user) it is possible to implement arc deletion and garbage collection
00089 //    in an efficient (relatively) manner. For the time being we haven't seen an
00090 //    application for this.
00091 //
00092 // The ForwardEbertGraph representation is like the general case described
00093 // above, with the following modifications:
00094 //  * The part of the head_[] array with negative indices is absent. In its
00095 //    place is a pointer tail_ which, if assigned, points to an array of tail
00096 //    nodes indexed by (nonnegative) arc index. In typical usage tail_ is NULL
00097 //    and the memory for the tail nodes need not be allocated.
00098 //  * The array of arc tails can be allocated as needed and populated from the
00099 //    adjacency lists of the graph.
00100 //  * Representing only the forward star of each node implies that the graph
00101 //    cannot be serialized directly nor rebuilt from scratch from just the head_
00102 //    array. Rebuilding from scratch requires constructing the array of arc
00103 //    tails from the adjacency lists first, and serialization can be done either
00104 //    by first constructing the array of arc tails from the adjacency lists, or
00105 //    by serializing directly from the adjacency lists.
00106 //  * The memory consumption is: m * sizeof(NodeIndexType)
00107 //                             + m * sizeof(ArcIndexType)
00108 //                             + n * sizeof(ArcIndexType)
00109 //    plus a small constant when the array of arc tails is absent. Allocating
00110 //    the arc tail array adds another m * sizeof(NodeIndexType).

01216   void BuildRepresentation() {
01217     first_incident_arc_.SetAll(kNilArc);
01218     DCHECK(TailArrayComplete());
01219     for (ArcIndexType arc = kFirstArc; arc <  max_num_arcs_; ++arc) {
01220       DCHECK(CheckTailIndexValidity(arc));
01221       Attach((*tail_)[arc], arc);
01222     }

01224   }

01226   bool BuildTailArray() {
01227     // If (*tail_) is already allocated, we have the invariant that
01228     // its contents are canonical, so we do not need to do anything
01229     // here in that case except return true.
01230     if (tail_ == NULL) {
01231       if (!representation_clean_) {
01232         // We have been asked to build the (*tail_) array, but we have
01233         // no valid information from which to build it. The graph is
01234         // in an unrecoverable, inconsistent state.
01235         return false;
01236       }
01237       // Reallocate (*tail_) and rebuild its contents from the
01238       // adjacency lists.
01239       tail_.reset(new ZVector<NodeIndexType>);
01240       tail_->Reserve(kFirstArc, max_num_arcs_ - 1);
01241       typename Base::NodeIterator node_it(*this);
01242       for (; node_it.Ok(); node_it.Next()) {
01243         NodeIndexType node = node_it.Index();
01244         OutgoingArcIterator arc_it(*this, node);
01245         for (; arc_it.Ok(); arc_it.Next()) {
01246           (*tail_)[arc_it.Index()] = node;
01247         }
01248       }
01249     }
01250     DCHECK(TailArrayComplete());
01251     return true;
01252   }

01287   // Reserves space for the arrays indexed by arc indices, except
01288   // (*tail_) even if it is present. We cannot grow the (*tail_) array
01289   // in this method because this method is called from
01290   // Base::Reserve(), which in turn is called from the base template
01291   // class constructor. That base class constructor is called on *this
01292   // before this->tail_ is constructed. Hence when this method is
01293   // called, this->tail_ might contain garbage. This method can safely
01294   // refer only to fields of the base template class, not to fields of
01295   // *this outside the base template class.
01296   //
01297   // The strange situation in which this method of a derived class can
01298   // refer only to members of the base class arises because different
01299   // derived classes use the data members of the base class in
01300   // slightly different ways. The purpose of this derived class
01301   // method, then, is only to encode the derived-class-specific
01302   // conventions for how the derived class uses the data members of
01303   // the base class.
01304   //
01305   // To be specific, the forward-star graph representation, lacking
01306   // reverse arcs, allocates only the positive index range for the
01307   // head_ and next_adjacent_arc_ arrays, while the general
01308   // representation allocates space for both positive- and
01309   // negative-indexed arcs (i.e., both forward and reverse arcs).
01310   void ReserveNextAdjacentArcAndHeadEntries(ArcIndexType new_max_num_arcs) {
01311     head_.Reserve(kFirstArc, new_max_num_arcs - 1);
01312     next_adjacent_arc_.Reserve(kFirstArc, new_max_num_arcs - 1);
01313     for (ArcIndexType arc = max_num_arcs_; arc < new_max_num_arcs; ++arc) {
01314       head_.Set(arc, kNilNode);
01315       next_adjacent_arc_.Set(arc, kNilArc);
01316     }
01317   }
01318 
01319   // Reserves space for the (*tail_) array.
01320   //
01321   // This method is separate from
01322   // ReserveNextAdjacentArcAndHeadEntries() because our practice of
01323   // making the (*tail_) array optional implies that the tail_ pointer
01324   // might not be constructed when the
01325   // ReserveNextAdjacentArcAndHeadEntries() method is
01326   // called. Therefore we have this method also, and we ensure that it
01327   // is called only when tail_ is guaranteed to have been initialized.
01328   void ReserveTailArray(ArcIndexType new_max_num_arcs) {
01329     if (tail_ != NULL) {
01330       // The (*tail_) values are already canonical, so we're just
01331       // reserving additional space for new arcs that haven't been
01332       // added yet.
01333       if (tail_->Reserve(kFirstArc, new_max_num_arcs - 1)) {
01334         for (ArcIndexType arc = tail_->max_index() + 1;
01335              arc < new_max_num_arcs;
01336              ++arc) {
01337           tail_->Set(arc, kNilNode);
01338         }
01339       }
01340     }
01341   }

01343   void ReserveInternal(NodeIndexType new_max_num_nodes,
01344                        ArcIndexType new_max_num_arcs) {
01345     ReserveTailArray(new_max_num_arcs);
01346   }

01489 // Builds a directed line graph for 'graph' (see "directed line graph" in
01490 // http://en.wikipedia.org/wiki/Line_graph). Arcs of the original graph
01491 // become nodes and the new graph contains only nodes created from arcs in the
01492 // original graph (we use the notation (a->b) for these new nodes); the index
01493 // of the node (a->b) in the new graph is exactly the same as the index of the
01494 // arc a->b in the original graph.
01495 // An arc from node (a->b) to node (c->d) in the new graph is added if and only
01496 // if b == c in the original graph.
01497 // This method expects that 'line_graph' is an empty graph (it has no nodes
01498 // and no arcs).
01499 // Returns false on an error.
01500 template<typename GraphType> bool BuildLineGraph(const GraphType& graph,
01501                                                  GraphType* const line_graph) {
01502   if (line_graph == NULL) {
01503     LOG(DFATAL) << "line_graph must not be NULL";
01504     return false;
01505   }
01506   if (line_graph->num_nodes() != 0) {
01507     LOG(DFATAL) << "line_graph must be empty";
01508     return false;
01509   }
01510   typedef typename GraphType::ArcIterator ArcIterator;
01511   typedef typename GraphType::OutgoingArcIterator OutgoingArcIterator;
01512   // Sizing then filling.
01513   typename GraphType::ArcIndex num_arcs = 0;
01514   for (ArcIterator arc_iterator(graph);
01515        arc_iterator.Ok();
01516        arc_iterator.Next()) {
01517     const typename GraphType::ArcIndex arc = arc_iterator.Index();
01518     const typename GraphType::NodeIndex head = graph.Head(arc);
01519     for (OutgoingArcIterator iterator(graph, head);
01520          iterator.Ok();
01521          iterator.Next()) {
01522       ++num_arcs;
01523     }
01524   }
01525   line_graph->Reserve(graph.num_arcs(), num_arcs);
01526   for (ArcIterator arc_iterator(graph);
01527        arc_iterator.Ok();
01528        arc_iterator.Next()) {
01529     const typename GraphType::ArcIndex arc = arc_iterator.Index();
01530     const typename GraphType::NodeIndex head = graph.Head(arc);
01531     for (OutgoingArcIterator iterator(graph, head);
01532          iterator.Ok();
01533          iterator.Next()) {
01534       line_graph->AddArc(arc, iterator.Index());
01535     }
01536   }
01537   return true;
01538 }
